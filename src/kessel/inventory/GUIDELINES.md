# Inventory Module Guidelines

Rules for working in `src/kessel/inventory/` -- the `ClientBuilder` base class, per-version wiring, and generated protobuf stubs.

## Directory Structure

```text
inventory/
  index.ts          # Hand-written. ClientBuilder abstract class + clientBuilderForStub() factory
  v1/
    index.ts        # Hand-written. Wires KesselInventoryHealthServiceClient
    health.ts       # Generated
  v1beta1/
    index.ts        # Hand-written. Wires multiple legacy service clients
    *               # All other files generated
  v1beta2/
    index.ts        # Hand-written. Wires KesselInventoryServiceClient
    __tests__/      # Tests for the builder
    *               # All other files generated
```

## Generated vs Hand-Written -- The Critical Distinction

Files with `// Code generated by protoc-gen-ts_proto. DO NOT EDIT.` are regenerated every 6 hours by CI. Never edit them. Changes to protobuf types must go through the upstream definitions at `buf.build/project-kessel/inventory-api`.

The only hand-written files in version directories are `index.ts` files.

## ClientBuilder Pattern

`index.ts` (this directory's root) exports:

- `ClientBuilder<T>` -- abstract class with fluent auth methods
- `clientBuilderForStub()` -- factory that creates a concrete builder from a generated service client class

### How per-version wiring works

Each version's `index.ts` calls `clientBuilderForStub(GeneratedServiceClient)` and exports the result:

```typescript
// v1beta2/index.ts
import { KesselInventoryServiceClient } from "./inventory_service";
import { clientBuilderForStub } from "..";
export const ClientBuilder = clientBuilderForStub(KesselInventoryServiceClient);
```

- v1 and v1beta2 export a single `ClientBuilder` const.
- v1beta1 exports multiple named builders (one per service). This is legacy and NOT in package exports.

### Adding a new API version

1. Run `buf generate` -- if the proto exists upstream, files appear automatically.
2. Create a hand-written `index.ts` that calls `clientBuilderForStub(GeneratedClient)`.
3. Add `package.json` exports entries (require/import/types triple + wildcard).
4. Add tests in `__tests__/index.ts` mirroring the v1beta2 test patterns.

## Builder Authentication Methods

| Method                                            | Channel       | Call Creds | Usage            |
| ------------------------------------------------- | ------------- | ---------- | ---------------- |
| `.insecure()`                                     | Insecure      | None       | Local dev only   |
| `.unauthenticated(channelCreds?)`                 | SSL (default) | None       | TLS without auth |
| `.oauth2ClientAuthenticated(auth, channelCreds?)` | SSL (default) | OAuth2     | Production       |
| `.authenticated(callCreds?, channelCreds?)`       | SSL (default) | Custom     | Custom auth      |

- Default (no method called): uses `credentials.createSsl()`.
- Last auth method called wins -- they are not additive.
- Combining call credentials with insecure channel throws at configuration time.

## build() vs buildAsync()

- `build()` returns the raw `@grpc/grpc-js` callback-based client.
- `buildAsync()` wraps it with `promisifyClient()` -- unary methods become Promise-based.
- Streaming methods (`streamedListObjects`, `streamedListSubjects`) are NOT promisified -- they already return `AsyncIterable`.

## Credential Validation

`validateCredentials()` is called synchronously in every auth method. It throws `"Invalid credential configuration: can not authenticate with insecure channel"` when call credentials are combined with an insecure channel. This is a build-time check, not a call-time check.

## Testing Conventions

Tests are in `v1beta2/__tests__/index.ts`. They do NOT start a real gRPC server. They verify:

- Builder construction and method presence (`typeof client.method === "function"`)
- Fluent API chaining returns without throwing
- Credential validation throws on invalid combinations
- `build()` and `buildAsync()` produce different instances
- Promisified methods have fewer parameters than callback versions

## Package Exports

When adding new public modules, update `package.json` `exports` with:

- A named entry with `require`, `import`, and `types` conditions
- A wildcard entry for deep imports (e.g., `./kessel/inventory/v1/*`)

v1beta1 is intentionally excluded from exports -- it is internal/legacy.

## Do Not

- Edit any file with `// Code generated by protoc-gen-ts_proto. DO NOT EDIT.`
- Import from package subpaths in source code -- use relative imports
- Create multiple client instances per service -- reuse a single instance
- Assume streaming methods are promisified by `buildAsync()`
